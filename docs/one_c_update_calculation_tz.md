# UpdateCalculation — API для порталу

Оновлення прорахунку в 1С до моменту створення замовлень. Документ у 1С оновлюється, новий не створюється, `calculationGUID` не змінюється.

## Виклик
- Метод: `POST` (та сама адреса HTTP-сервісу, що й `CreateCalculation`)
- Заголовок: `Query: UpdateCalculation`
- Тіло: `JSON`

1С обробляє перший елемент масиву `calculations` — надсилайте один прорахунок за запит.

## Payload

| Поле | Тип | Обов'язкове | Опис |
| --- | --- | --- | --- |
| `calculationGUID` | `string (uuid)` | так | Ідентифікатор прорахунку, отриманий з відповіді `CreateCalculation` |
| `calculationNumber` | `string` | ні | Номер прорахунку дилера |
| `itemsCount` | `number` | ні | Кількість конструкцій |
| `comment` | `string` | ні | Коментар |
| `address` | `object` | ні | Адреса доставки, та сама структура, що й у `CreateCalculation` |
| `recipient` | `object` | ні | Отримувач: `recipientName`, `recipientPhone`, `recipientAddionalInformation` |
| `file` | `array` | ні | Лише змінені або нові файли |
| `updatedAt`, `kontragentGUID`, `authorGUID`, `orders` | — | ні | Приймаються, але ігноруються — контрагент і автор не змінюються |

Головний принцип: надсилайте лише те, що користувач змінив. Поле, якого немає в запиті, у 1С залишається без змін.

## Правила для `file`

Поле `file` передавайте тільки якщо користувач замінив основний файл або додав нові фото. Якщо файли не змінювались — не передавайте `file` взагалі, вкладення в 1С збережуться.

- `fileDataType: "Calculation"` — новий основний файл прорахунку. 1С помітить старий файл на видалення і збереже новий. Передавайте не більше одного.
- `fileDataType: "Photo"` — тільки нові фото. 1С додає їх до вже збережених, наявні фото не видаляються і повторно їх надсилати не потрібно.
- Видалення або заміна окремих фото через цю точку не підтримується.

## Приклади

### Зміна лише коментаря, без файлів

```json
{
  "calculations": [
    {
      "calculationGUID": "ffc5b842-7f7e-11f1-9ced-4cd98f08e56d",
      "comment": "Уточнений коментар"
    }
  ]
}
```

### Заміна основного файла і одне нове фото

```json
{
  "calculations": [
    {
      "calculationGUID": "ffc5b842-7f7e-11f1-9ced-4cd98f08e56d",
      "itemsCount": 3,
      "file": [
        {
          "fileName": "calc.zkz",
          "fileDataB64": "...",
          "fileExtension": "ZKZ",
          "fileDataType": "Calculation"
        },
        {
          "fileName": "photo_2.jpg",
          "fileDataB64": "...",
          "fileExtension": "JPG",
          "fileDataType": "Photo"
        }
      ]
    }
  ]
}
```

## Відповіді

HTTP-статус завжди `200`, результат — у тілі відповіді.

### Успіх

```json
{
  "success": true,
  "results": [
    {
      "calculationGUID": "uuid"
    }
  ]
}
```

### Помилка

```json
{
  "success": false,
  "error": "текст помилки"
}
```

## Типові `error` і реакція порталу

| `error` | Що робити на порталі |
| --- | --- |
| `Редагування неможливе: за прорахунком уже створені замовлення.` | Заблокувати редагування прорахунку, показати повідомлення користувачу |
| `Прорахунок не знайдено` | Прорахунку немає в 1С — перевірити GUID |
| `Прорахунок помічено на видалення` | Прорахунок видалено в 1С |
| `Не передано calculationGUID`, `Невірний формат calculationGUID: ...` | Помилка формування запиту |
| `Помилка: Неможливо прочитати JSON!`, `Помилка: нема даних в JSONі!` | Некоректне тіло запиту |
| інший текст | Помилка запису в 1С — показати і залогувати текст |

## Порада

Перевірка замовлень відбувається на боці 1С у момент запиту, тому навіть якщо портал вважає, що замовлень немає, потрібно обробляти відповідь `success: false` — замовлення могли створити в 1С щойно.
